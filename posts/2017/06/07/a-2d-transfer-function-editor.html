A 2-D Transfer Function Editor

Jun 7, 2017

In my previous post, I talked about visualizing 3D scalar fields using isosurfaces. A significant flaw with rendering as isosurfaces is that different possibilities (or rather regions of the surface) cannot be seen simultaneously as we observe the surface only for the selected isovalue. In addition to this, an isosurface is assigned the same color and opacity values to every point on the isosurface obscuring internal regions. Volume Rendering is a technique which resolves this by making use of transfer functions.

Volume Rendering

Volume Rendering (sometimes abbreviated as VR) is a direct visualization technique (in contrat to Isosurfaces), in that it tries to construct the surface directly from the data (rather than finding the triangles). An example of a VR method is raycasting, which simulates rays passing through the data and then reconstructs the pixels based on ray intersections.

Transfer function

In simple terms, a transfer function maps a scalar value to a color-opacity pair . This is a 1-D transfer function since it takes only the scalar values into the account. Typically, a user assigns for a value, although there has been research on automated transfer functions. To discover interesting properties of a surface requires building a effective transfer function which in most cases requires a great deal of user interaction. A 1-D transfer function still has issues with ambiguous boundaries (lack of edge detection information). Interface

A brief illustration of the interface and the interactions associated with it is shown below

There are 3 key widgets - one panel with a (log scaled) histogram of the scalar values versus the opacity on the Y-axis, one panel with the colormap ranging horizontally across the scalar values and finally a panel with a histogram (log scaled) of the gradient magnitudes versus Opacity. All the 3 widgets have capability to add “control points” - a mapping for the scalar/gradient value at that point to the color/opacity.

The window on the side renders the volume using VTK’s openGL rendering pipeline. Each dimension (scalar value/gradient magnitude) corresponds to a vtkPieceWiseFunction object (docs here). The vtkVolumeProperty (docs here) class can be used to set the color/opacity for a particular point.

For the GUI widgets, I’ve used ofxImGui which is an openFrameworks addon for the neat C++ UI library ImGui.

The dataset being shown is for a car engine and we can get to see the interior parts of it by adding control points to the panels and appropriately adjusting the opacity by dragging them along the histogram.

The rendering is fairly slow as only raycasting is supported in Linux by VTK.

Summary

While the 2D transfer function offers more options fo the user, this also can become cumbersome especially if lot of customization is required with regard to the widgets. This is the trade-off when we increase the dimensionality. Dimensionality also increases the computational intensiveness of rendering and interactivity as observed above. A 3D transfer function is also possible by including the second derivate of the scalar values.
